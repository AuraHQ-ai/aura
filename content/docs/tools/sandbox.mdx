# Sandbox (E2B)

Aura has access to a sandboxed Linux VM via [E2B](https://e2b.dev). This allows her to run arbitrary code, install packages, and execute complex multi-step operations safely.

## Available Tools

### run_command
Execute a shell command in the sandbox. Supports environment variables, working directories, and timeouts.

## How It Works

Each sandbox session provides:
- A full Linux environment (Ubuntu-based)
- Pre-installed: Node.js, Python, git, GitHub CLI
- Network access for installing packages and cloning repos
- File system persistence within a session
- Automatic cleanup after timeout

## Environment Variables

Environment variables are passed per-command, not persisted in the sandbox:

```typescript
await sandbox.commands.run("echo $API_KEY", {
  envs: { API_KEY: process.env.API_KEY },
});
```

This is deliberate — E2B sandboxes can be paused and resumed, and environment variables don't persist across pause/resume cycles.

## Use Cases

- **Code execution** — Run user-provided scripts, test code snippets
- **Data processing** — Transform files, parse documents, generate reports
- **Self-improvement** — Clone the repo, make changes, create branches, open PRs
- **Browser automation** — Run Playwright for web scraping and testing
- **Document conversion** — Parse PDFs, spreadsheets, and other file formats

## Safety

The sandbox is completely isolated from the Vercel runtime. It cannot access:
- The production database
- Slack tokens (unless explicitly passed via `envs`)
- Other environment variables (unless explicitly passed)

This isolation is the safety boundary — Aura can run untrusted code without risk to the production system.
